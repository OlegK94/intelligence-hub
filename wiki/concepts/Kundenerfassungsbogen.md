# Kundenerfassungsbogen (KEB)

## Overview

The **Kundenerfassungsbogen** (KEB, literally "customer registration form") is [[DoktorLib]]'s formal document capturing all key customer data required for the deployment (Deployment) of their Praxissoftware at an Arztpraxis (medical practice). It is the central handover artifact from Sales to the Deployment team.

## Current State (Problem)

- **Format:** Manually filled Excel spreadsheet
- **Process:** Sales rep fills in data from multiple touchpoints (demo call, discovery notes, IT audit)
- **Problems:**
  - Error-prone (manual entry)
  - Not a single source of truth
  - Deployment team works from a static file (no live updates)
  - Disconnected from Salesforce and Notion

## Future State (Target)

With the [[DoktorLib Automation Pipeline]]:

```
Vertragsunterzeichnung (trigger)
    ↓
[DUST AGENT] reads all Salesforce data
    ↓
KEB generated automatically in standard format
    ↓
Deployment accesses via Notion link (no Excel)
```

- **Format:** Auto-generated by [[Dust Agent]] from [[Salesforce]] data
- **Access:** Deployment team receives a [[Notion]] link
- **Source of Truth:** Single, live, connected record

## KEB Field Structure

As of source document (2026-05-08): **noch zu definieren** (not yet defined). Field definition is a prerequisite for the Dust Agent prompt — cannot be written without knowing what the KEB must contain.

Expected field categories based on Tally intake fields:
- Praxistyp
- Fachrichtung
- Aktuelle Software
- Labore
- IT-Infrastruktur
- Gerätelandschaft
- Discovery Notes (from demo call)
- IT Audit data (from IT partner)

## Strategic Significance

The KEB is the **artifact that defines what data must be collected** throughout the entire pipeline. Its field structure determines:
- What Tally intake form must ask
- What the IT Audit Tally form must capture
- What the Dust Agent prompt must generate

This is why the source recommends **defining Tally fields first** — the KEB output structure should drive the input structure.
